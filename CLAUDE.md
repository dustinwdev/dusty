# Dusty

Reactive, declarative GUI framework for Rust. Desktop-first. Inspired by React's component model, Solid.js's fine-grained reactivity, and Tailwind's utility styling.

## Architecture

Layered crate workspace — each layer is a separate crate with clean trait boundaries:

| Crate | Purpose | Key Deps |
|-------|---------|----------|
| `dusty` | Facade — re-exports everything | all below |
| `dusty-reactive` | Signals, memos, effects, resources | none (pure Rust) |
| `dusty-core` | View trait, node types, events | dusty-reactive |
| `dusty-style` | Utility styling + theme engine | dusty-reactive |
| `dusty-layout` | Flexbox layout | taffy |
| `dusty-widgets` | Built-in widget library | dusty-core, dusty-style |
| `dusty-text` | Text shaping/rendering | cosmic-text |
| `dusty-a11y` | Accessibility tree | accesskit |
| `dusty-render` | Scene graph + GPU rendering | wgpu |
| `dusty-platform` | Windowing, input, event loop | winit |
| `dusty-macros` | `#[component]`, `col![]`, etc. | syn, quote |
| `dusty-devtools` | Inspector, profiler | dusty-core |

Dependency flow: reactive → core → style/layout → widgets → render/platform → facade.

## Workflow

### TDD (Strict)

Every feature follows **Red → Green → Refactor**:

1. **Red** — Write failing tests that define expected behavior and API surface
2. **Green** — Minimal implementation to pass
3. **Refactor** — Clean up, optimize, keep tests green

No implementation code without a test first. Tests ARE the spec.

### Checklists

Every crate or significant feature gets a checklist in `docs/checklists/`. Workflow:

1. Create checklist before writing code — define scope, API surface, test cases
2. Work through items sequentially
3. Mark items done with `[x]` as completed
4. Update checklist as scope clarifies — they are living documents

### Documentation

Internal working docs live in `docs/`:

- `docs/design/` — architecture decisions, API sketches, trade-off analysis
- `docs/checklists/` — implementation checklists per crate/feature

These are working documents for us, not public docs. Keep them practical and current. Delete or update stale content.

## Coding Standards

### Rust

- `cargo fmt` — always, no exceptions
- `cargo clippy -- -D warnings` — zero warnings
- No `unwrap()` or `expect()` in library code — use `Result` with structured errors via `thiserror`
- `unsafe` requires a `// SAFETY:` comment explaining the invariant
- Public API items get doc comments; include a short example on key types/functions
- Prefer `impl Trait` in argument position for ergonomic APIs
- Keep crate APIs minimal — pub only what's needed, `pub(crate)` by default

### Testing

- Unit tests: `#[cfg(test)] mod tests` in each module
- Integration tests: `tests/` directory per crate
- Property-based tests with `proptest` where applicable (especially reactive, layout)
- Test names describe behavior: `signal_notifies_subscribers_on_change`
- Test edge cases: empty state, rapid updates, concurrent access, drop ordering

### Commits

- Conventional commits: `feat:`, `fix:`, `test:`, `refactor:`, `docs:`, `chore:`
- Small, focused — one concern per commit
- **No AI attribution** — never add `Co-Authored-By`, `Generated by`, or any other AI/LLM credit to commit messages, PR descriptions, or any git metadata

## Running

```bash
cargo test --workspace          # all tests
cargo test -p dusty-reactive    # single crate
cargo clippy --workspace -- -D warnings
cargo fmt --all -- --check
```
